# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

Six sub-projects deliver a unified "mini-OS" desktop experience on myhost (`192.168.1.10`), exposed publicly at `https://service.example.com/` via Cloudflare Tunnel with Access auth. The root page (`/`) is a desktop-like UI launchable from a Start menu with nine everyday apps (Services, Terminal, Browser, X11 Launcher, Files, Office, Notes, Monitor, Upload) plus an **Update** app in a separate "System" section.

| Sub-project | URL path | What |
|---|---|---|
| terminal | `/t1/`..`/t50/`, `/terminals/`, `/api/` | Dynamic persistent bash terminals (ttyd + vibetop-session) + manager API |
| browser | `/browser/`, `/apps-display/` | Persistent Chromium via xpra HTML5 (`:99`) + a second xpra display (`:98`) for the X11 Launcher's GUI apps |
| landing | `/` | Unified desktop UI with tab bar, iframe viewport, and status bar |
| files | `/files/` | FileBrowser file manager rooted at `/` (whole filesystem, as `APP_USER`) |
| office | `/onlyoffice/` | OnlyOffice Document Server (Docker) ŌĆö in-browser Office editing via the manager's `/api/office/*` |
| claude-usage | `/api/claude/usage` | Opt-in pass-through proxy that captures the real Claude Max-plan usage headers for the desktop's usage strip |
| tunnel | ŌĆö | Cloudflare Tunnel + Access config for public HTTPS |

## Deploy commands

**Fresh host, one line** ŌĆö `bootstrap.sh` is the curl-pipe installer: it checks
the OS is Debian/Ubuntu, installs `git`, clones (or `git`-updates) the repo to
`~/vibetop` (full clone so the in-app Updater works), then `exec`s `deploy.sh`.
It is the only step `deploy.sh` can't do itself ŌĆö getting the repo onto the box.
Runs as a normal sudo user (it refuses root: the desktop runs as `APP_USER` and
`landing/install.sh` would deploy to `/root`). Flags after `-s --` pass through
to `deploy.sh`; env overrides `VIBETOP_DIR`/`VIBETOP_REPO`/`VIBETOP_REF`.

```bash
curl -fsSL https://raw.githubusercontent.com/nicejunjie/vibetop/main/bootstrap.sh | bash
curl -fsSL .../bootstrap.sh | bash -s -- --no-office   # forward deploy.sh flags
```

**One command, whole stack** ŌĆö `deploy.sh` orchestrates everything (deps + all
sub-installers in the right order + a health check), locally or to a remote host.
It also **auto-detects a dual-homed LAN** (2+ NICs on one subnet) and applies the
"reply via the incoming NIC" routing (`tools/setup-samesubnet-routing.sh`) so such
hosts work with no manual network tweaking ŌĆö a no-op on single-homed hosts (see the
dual-homed gotcha + `docs/dual-homed-network.md`):

```bash
./deploy.sh                                  # deploy on this machine
./deploy.sh --remote junjie@192.168.1.20     # rsync to HOST:~/vibetop and deploy there
# flags: --no-browser  --no-files  --no-office  --with-tunnel  --dry-run
# (HOST is any ssh destination ŌĆö user@ip or an ssh-config Host, not a bare shell alias)
sudo ./uninstall.sh                          # tear down the whole runtime (keeps repo + data + image)
```

It is fully self-installing on a Debian/Ubuntu host (incl. Docker) ŌĆö no manual
prerequisites. Or run the per-project installers by hand (the order `deploy.sh`
uses). Each is idempotent, supports `--dry-run`, env-var configurable (see script
headers), and **only reloads nginx when its config actually changed** (a re-run
that changes nothing won't reload ŌĆö which would otherwise sever live terminal/
Browser WebSockets; `nginx_write` returns the change as its pipe exit status):

```bash
sudo ./terminal/install.sh   # 1. nginx site skeleton (extras include) + manager API + ttyd
sudo ./browser/install.sh    # 2. xpra + Chromium (snap) + LibreOffice (office View) ŌĆö extras snippet
sudo ./files/install.sh      # 3. FileBrowser at /files/ (binary + noauth config + extras snippet)
sudo ./office/install.sh     # 4. Docker + OnlyOffice Document Server at /onlyoffice/ (office Edit)
./landing/install.sh         # 5. desktop UI + static apps (no sudo ŌĆö $HOME must resolve to the user's)
sudo ./tunnel/install.sh     # 6. cloudflared (tunnel setup is interactive ŌĆö see tunnel/README.md)
```

Deps the installers handle automatically: `ttyd`/`nginx`/`acl` (apt), `xpra` (xpra.org
apt repo, suite derived from the OS codename) + `chromium` (snap) + `libreoffice`
(apt), the `filebrowser` release binary (pinned `FB_VERSION`, arch-aware), and
**Docker** (`docker.io`) running `onlyoffice/documentserver` (~2 GB pull, loopback
`:8087`, generated JWT secret at `~/.config/vibetop/onlyoffice.secret`). Scoped to
Debian/Ubuntu. Validated on AMD+NVIDIA and AMD+AMD hosts (GPU stats use
sysfs/amdgpu with an `nvidia-smi` fallback).

## Tests

**Python** ŌĆö unit/smoke tests for the manager's security-critical and pure logic
live in `terminal/tests/` (pytest). They run without root or any of the systemd/
nginx/Docker stack ŌĆö `conftest.py` loads the hyphenated `terminal-manager.py` via
`importlib` and puts `terminal/` on `sys.path` so its `import system_status`
resolves:

```bash
cd terminal && python -m pytest tests/ -q
```

Coverage targets the things where a silent regression is dangerous: the
shell-injection guard (`_valid_browser_url`), path-traversal guard
(`_resolve_under_home`/`OFFICE_RE`), hand-rolled JWT/HMAC (`_jwt_*`,
`_onlyoffice_sig`, incl. an `alg:none` forgery test), the streaming multipart
parser, upload-name sanitization, atomic writes, the **cross-instance desktop
state machine** (`test_desktop.py` ŌĆö `_desktop_union`/`_desktop_prune_targets`/
`_desktop_cap` TTL + close-target math), the **`/api/ping` + `/api/metrics`**
counters (`test_metrics.py` boots the server in-thread over a real socket), the
**ring-buffer replay sanitization** (`test_claude_session.py` ŌĆö
`strip_terminal_queries` drops stale DA/CPR/color/mode *probes* from a reconnect
replay so they aren't re-answered into the prompt, while preserving real screen
state), the `system_status` collector, and the **Claude-usage settings surgery**
(`test_claude_usage.py` ŌĆö `_set_claude_usage_env` preserves the user's other
settings and only removes *our* `ANTHROPIC_BASE_URL` on disable; the toggle never
runs `--now` on disable so a pinned session's proxy isn't stopped), and the
**service-discovery parser** (`test_service_discovery.py` ŌĆö `parse_ss` non-loopback
extraction, `classify` port/proc denylists, and `_effective_proc` unmasking a
daemon that listens under a generic `python3`). Prefer adding a test here when
touching any.

**JavaScript** ŌĆö the fragile front-end logic that kept regressing is now DOM-free
and unit-tested with node's built-in runner (no deps):

```bash
node --test terminal/lib/    # tab-set reconciliation (tab-sync.js)
node --test landing/         # service-worker fetch-routing rules (sw.js)
```

`terminal/lib/tab-sync.js` is the pure tab-set reconcile/`nextAvailable` math
lifted out of `terminals.html` (which loads it via `<script src>`, content-hash
cache-busted by `install.sh` like `terminal-kbd.js`) ŌĆö its tests pin the
open/close/poll-lag race cases behind the v1.9.x churn. `landing/sw.test.js`
parses the **live** `BYPASS` regex + `PRECACHE` out of `sw.js` (so it tracks the
real source, no drift) and asserts every path classifies correctly: bypass vs.
cacheable-shell-nav vs. network-only-nav vs. SWR sub-resource.

A focused **security review** of the manager's auth paths (no vulns found ŌĆö the
trust model is the takeaway) is in `docs/security-review.md`.

**CI + pre-commit** ŌĆö `.github/workflows/tests.yml` runs both suites on every
push/PR (no services/root: the Python tests load the manager in-process with the
one systemctl call stubbed; the JS tests are dep-free). A versioned pre-commit
hook (`.githooks/pre-commit`) runs the same suites locally ŌĆö enable it once per
clone:

```bash
git config core.hooksPath .githooks      # then commits run pytest + node --test
```

Bypass a single commit with `git commit --no-verify` or `SKIP_TESTS=1 git commit`;
each runner self-skips if its tool isn't installed.

## Health check

```bash
systemctl status vibetop-manager vibetop-browser-xpra vibetop-apps-xpra vibetop-apps-dbus vibetop-filebrowser
docker ps --filter name=vibetop-onlyoffice                      # OnlyOffice container (office Edit)
curl -sI http://127.0.0.1/ http://127.0.0.1/t1/ http://127.0.0.1/browser/ http://127.0.0.1/apps-display/ http://127.0.0.1/files/
curl -s http://127.0.0.1/api/events --max-time 2 | head -1      # SSE auto-refresh stream (-> "retry: 5000")
curl -s http://127.0.0.1/onlyoffice/healthcheck                 # -> true when the doc server is up
curl -s http://127.0.0.1/api/system/status
curl -s http://127.0.0.1/api/terminals/status
curl -s http://127.0.0.1/api/ping                               # -> {"ok":true} (liveness; what the systemd watchdog probes)
curl -s http://127.0.0.1/api/metrics                            # self-metrics: requests/latency/errors, SSE clients, terminal churn, uptime
sudo systemctl status cloudflared
sudo tail -f /var/log/vibetop/manager.log                       # manager actions/errors (also: journalctl -u vibetop-manager)
```

The manager unit has a **liveness watchdog**: `WatchdogSec=60` in
`vibetop-manager.service` + a thread (`_watchdog_loop`) that pets it only while
a loopback `GET /api/ping` actually answers ŌĆö so a *wedged* manager (stuck accept
loop, exhausted threads) is restarted, which the crash-only `Restart=on-failure`
would miss. `_sd_notify` is a dependency-free sd_notify (no-op without
`NOTIFY_SOCKET`, so local/test runs are unaffected).

**Backups** ŌĆö `tools/backup.sh` tars the irreplaceable host-local state
(`~/.local/share/desktop-*`, terminal/notes/update data, FileBrowser DB, the
OnlyOffice secret, `~/Documents`) to `~/vibetop-backups`, keeping the newest 14.
`--dry-run` previews, `--install-timer` sets up a daily systemd timer, `--list`/
`--restore FILE` manage archives. None of this state is in git, so it's the one
thing a disk loss would take.

The manager logs via a `vibetop` Python logger to **both** journald (stderr) and a
self-rotating file `/var/log/vibetop/manager.log` (`RotatingFileHandler`,
~12 MB cap = 2 MB ├Ś 6, no logrotate/cron needed). It's **selective**: `INFO` on
real actions (terminal start/stop, `x/launch`, cross-device close, reset, update
outcome, SSE reload push) and `WARNING` on failures; the per-request HTTP access
log is at `DEBUG` (off by default). Raise verbosity with `LOG_LEVEL=DEBUG` in the
unit's environment. See `docs/design-decisions.md` for the rationale.

## Architecture

### Unified desktop (`landing/desktop.html`)

The root page at `/` is a Windows-style shell:
- **Start button** ŌĆö always present at the taskbar's left. Clicking it toggles the **Start menu**, a launcher listing nine everyday apps (Services, Terminal, Browser, X11 Launcher, Files, Office, Notes, Monitor, Upload), then a separated **System** subsection with the **Update** app (set apart since it's rarely used), each with icon + description. Picking one opens it. (This replaced the old always-pinned tab bar.) **Services** is an **auto-discovering network-service dashboard** (`landing/index.html`, served at `/landing.html`) ŌĆö it lists the host's other web services (Ollama/Open WebUI, ThinLinc, Streamlit apps, Jupyter, ŌĆ”) with **zero config**. The manager's `GET /api/services/discover` (sibling module `terminal/service_discovery.py`) scans listening TCP sockets (`ss -tlnp`), keeps those bound to a **non-loopback** address (a process on `0.0.0.0`/`::` is a network service; loopback-only sockets ŌĆö incl. all of vibetop's own internals ŌĆö are excluded), drops infra ports/daemons (ssh/smtp/dns/rpc/db + vibetop's own `:80`), and **names each from its `/proc/<pid>/cmdline`** (a recognizer table for well-known apps ŌåÆ a port hint ŌåÆ `proc :port`; the *effective* program is pulled from the cmdline when the socket owner is a generic interpreter like `python3`, so `python3 ŌĆ”/vsmagent` is filtered and `python3 ŌĆ”/open-webui` is named "Open WebUI"). Each card's dot is green (it's listening); tapping a card opens the service in the **embedded Browser** (Chromium runs on the host, so it reaches the LAN service whether you're on the LAN or coming through the tunnel), with a `ŌåŚ new tab` link for LAN clients. Data comes from a direct `/api/services/discover` fetch (LAN) or a parent `postMessage` relay (tunnel). This replaced the old hand-maintained, gitignored `services.json` (which in practice stayed the shipped example ŌĆö hence the rebuild). Result memoized ~5s server-side.
- **Taskbar apps** ŌĆö only *opened* apps get a button (Windows-style), each with a close (├Ś). Multiple apps can be open at once; the focused one is highlighted with a per-app accent underline. Buttons are **drag-reorderable** (HTML5 DnD ŌĆö desktop only; it's a no-op on touch so tap/scroll are unaffected): dragover reshuffles the DOM live, dragend reads the new order back into `openApps` and persists it. App-switching is wired on **`pointerdown`** for mouse (close ├Ś and touch stay on `click`) for the same reason as the terminal tabs ŌĆö a focused app iframe makes the browser eat the first taskbar click moving focus back to the parent, so click-based switching needed two clicks. The set of open apps, their order, and the active one are persisted **server-side** via `GET/POST /api/desktop` (file `~/.local/share/desktop-state.json`). **Windows are local to each instance** ŌĆö opening an app on the phone does NOT open a window on the computer (window mirroring was tried and removed: the Browser is a single shared-size xpra display, so a wide desktop client connecting would force desktop layout onto the phone, and windows silently appearing is jarring). What *is* shared across instances: (1) the Start-menu **"running" dots show the UNION** of apps open on any live instance (awareness), (2) a **logout/reset clears every instance**, and (3) **close-on-all-devices** ŌĆö a running app's Start-menu row shows a soft **ŌÅ╗ power button** (only when running; a real tap-target, confirmed) that closes that app on *every* client (the one place an app open only on another device is reachable). Each browser has a stable `INSTANCE_ID` (localStorage); the manager keeps a **per-instance registry** (`{instances:{id:{open,active,ts}}, reset_epoch}`) ŌĆö a 5s heartbeat (`pushDesktop`, plus a re-register on `visibilitychange`) keeps an instance in the "live" union (`DESKTOP_TTL` 120s ŌĆö must exceed browsers' ~60s background-tab timer throttling or an idle machine's dots wrongly go dark elsewhere), and the same id lets a device **restore its own windows** (`GET /api/desktop?instance=`) on reload, not the other device's. Closing via the taskbar ├Ś is local (drops from this instance's set; the dot goes dark once no live instance has it). To close across devices, the Start-menu Ō£Ģ calls `POST /api/desktop/close {app}`, which records the **live instances that currently have the app open** in `close_targets` (`{appId:[instanceId,ŌĆ”]}`); each instance closes the app when it sees its own id listed in a heartbeat reply, then reports an open-set without it and the server prunes it out (also pruned when an instance goes stale). Targeting by instance id (not a global flag/counter) means **reloading the holding device still closes it**, and a stuck holder can't poison the app for other devices. `/api/reset` bumps `reset_epoch`; every other instance sees it advance on its next heartbeat and tears its own desktop down (`clearAllLocal`). On a fresh desktop with nothing open the Start menu auto-opens.
- **App frames** ŌĆö each app is a full-viewport iframe, **created** on first open but only `src`-loaded on first activation (`loadIfNeeded`) so the inner content always measures the real viewport (otherwise xterm.js / FitAddon initialise at 0├Ś0 and the terminal renders truncated). Closed apps are **removed from the DOM** (true unload). All iframes carry `allow="clipboard-read; clipboard-write"` for cross-iframe clipboard.
- **Status bar** ŌĆö live system stats (CPU% + temp, MEM used/total, GPU% + temp, VRAM used/total) that **ride the 5s desktop heartbeat** (the `system` field on `/api/desktop`, rendered by `renderSysStats` in `onDesktopResp`) rather than a standalone poll ŌĆö one of the shell-tier polls **consolidated onto the heartbeat** (the taskbar stats and the Start-menu terminal-count badge, `terminals_running`, both folded in; the standalone `/api/system/status` + `/api/terminals/status` client loops were removed). Both are memoized server-side and collected **outside `_desktop_lock`**, so the fold doesn't lengthen the lock or duplicate work across clients. Rendered in a fixed-width CSS grid (`ch`-sized columns + `tabular-nums`) so labels don't shift sideways as values change digit-count. CPU temp from `k10temp` (Tctl), GPU temp from `amdgpu` (edge). When the GPU driver locks sysfs during heavy compute (EBUSY), util/temp/power **fall back to parsing `/sys/kernel/debug/dri/N/amdgpu_pm_info`** (manager already runs as root) so the numbers stay populated. The full 1/5/15-minute load average is shown only in the Monitor app's CPU card title. This readout is a **toggleable "System Stats" utility** (Start Ō¢Ė Utilities): when off, the CPU/GPU/MEM/VRAM block (and its taskbar separator) are hidden **and the server omits the `system` field** (so it isn't collected) ŌĆö the taskbar space goes back to app tabs. The preference is **shared across all clients**: it lives on the desktop state (`sys_stats`) and is set via `POST /api/desktop/ui {sysStats}`, converging on every client within ~5s through the heartbeat (`applyServerSysStats` in `onDesktopResp`); `localStorage` is only a fast-load cache so there's no flash before the first heartbeat. (Deliberately **not** folded onto the heartbeat: the X11-window auto-surface poll ŌĆö a `wmctrl` shell-out that could block ŌĆö and the Services relay, which runs an `ss`/`/proc` scan only while the Services app is open; keeping conditional/heavier work off the heartbeat critical path is the point of consolidating *within* a tier rather than globally.) The **taskbar layout** keeps the tabs (`.task-apps { flex:1 1 auto; overflow-x:auto }`) growing to fill and scrolling when narrow, while the stats block, Start button, and Logout stay fixed-width (`flex-shrink:0`) ŌĆö so a long tab list can never shove the stats off-screen or overlap them (the overlap bug that made the stats collide with the right-most tab).
- **Logout button** ŌĆö taskbar far right (ŌÅ╗ glyph). Clicking it does a full **fresh-start reset** then logs out: after a `confirm()`, it `POST`s `/api/reset` (stops every running terminal + its background processes, clears the saved desktop layout, drops in-memory office edit sessions, and resets the Browser to a blank Chromium) and only then navigates to `/cdn-cgi/access/logout`. The `href` stays as the no-JS fallback. Over the tunnel `/cdn-cgi/access/logout` is handled at Cloudflare's edge (its own signed-out page); on the **LAN** there's no Cloudflare in front so that path hits the origin ŌĆö nginx serves a friendly `loggedout.html` via `location = /cdn-cgi/access/logout` (terminal site) instead of a 404.
- **PWA / installable** ŌĆö `landing/manifest.json` (+ `apple-*` meta tags and an `apple-touch-icon` in `desktop.html`) makes "Add to Home Screen" launch the desktop **standalone** (no Safari chrome ŌĆö which is why the `100svh`/URL-bar juggling matters less once installed). `landing/sw.js` is a service worker that caches the shell + static app pages (`/`, `landing.html`, `notes.html`, `monitor.html`, `upload.html`, patches JS, icons) for instant cold loads. It is **deliberately conservative**: page loads (`navigate` requests) are **network-first with a 2.5s timeout** ŌĆö so Cloudflare Access redirects still work (an expired session returns a redirect that's passed through and never cached) while a *stalled* network falls back to the cached shell (the iOS-Safari-on-flaky-wifi case); static sub-resources are cache-first/stale-while-revalidate; and everything live or auth-sensitive (`/api`, `/browser`, `/tN`, `/terminals`, `/files`, `/fileview`, `/services.json`, `/cdn-cgi`) is **bypassed** entirely (network only, never cached). Bump `VERSION` in `sw.js` to invalidate the cache after a shell change; `activate` deletes old caches, and nginx serves `sw.js` `no-store` so the browser re-checks it. The SW uses `skipWaiting()`+`clients.claim()`. **Auto-refresh on deploy (push, not poll):** the manager serves an **SSE stream at `GET /api/events`** that watches the deployed `sw.js` VERSION and pushes a `reload` event to every connected client the moment it changes (one server-side file-check replaces all client polling; `X-Accel-Buffering: no` + ~18s pings keep it streaming through nginx/Cloudflare with no nginx config). On a `reload` event ŌĆö **or** when a reconnect's `hello` reports a different version than the one this page loaded with (the **self-heal**: covers a tab that was offline/suspended during the deploy, so a missed push isn't lost forever ŌĆö it refreshes the moment it reconnects/foregrounds) ŌĆö the client kicks `registration.update()` and does a **full `window.location.reload()` directly** (brief "Updating vibetopŌĆ”" notice), **not** via `controllerchange`: an *uncontrolled* page (after a hard reload, `hadController=false`) never fires that, which silently ate the refresh. One reload refreshes the *entire* desktop incl. every app iframe ŌĆö a deploy reaches all of vibetop with no per-app update logic; persistent state survives (terminals/Browser reconnect, notes autosave, Files/Notes tabs are server-side). (A separate `controllerchange` handler ŌĆö for the browser's *own* background SW checks ŌĆö keeps the `hadController` guard so the first install doesn't reload mid-load and interrupt a Terminal still connecting: "had to refresh a few times before I could type". `/api/events` logs `[events] pushed reload vŌĆ”->vŌĆ”` so a deploy propagating is visible in the journal. Bootstrap: a client must already be on the auto-refresh build for any of this to fire; older open tabs need one manual reload to get onto it.) Icons (and a multi-size `favicon.ico` at the web root) are generated by `landing/icons/generate-icons.py` (a 2├Ś2 launcher grid in the app accent colors); `desktop.html` carries `apple-touch-icon` + `rel="icon"` links. On iOS an installed PWA has its own cookie jar, so the first launch may require a one-time Access re-auth. **Over the tunnel the icons/manifest are behind Cloudflare Access**, so a browser's cookie-less favicon fetch gets the login page ŌåÆ no favorite icon; fix with an Access **Bypass** app for `/favicon.ico`, `/apple-touch-icon.png`, `/manifest.json`, `/icons/*` (see `tunnel/README.md`).
- Relays `/api/services/discover` to the Services app via `postMessage`, only while it's open ŌĆö required for the Cloudflare tunnel where an iframe's own fetch doesn't carry the Access cookie, so the parent fetches and forwards. (Monitor fetches `/api/system/status` itself and uses no relayed data.)
- Listens for `postMessage` from the terminal app to open a clicked URL in the Browser app (launching Browser if it isn't already open). Also handles `open-files-at` (from the Upload app) to deep-link the Files app at a given path.
- **Mobile** ŌĆö viewport meta is pinned (`user-scalable=no`) since iOS pinch-zoom is page-wide and once stuck can't be reliably reset; each app handles its own zoom internally (xpra zooms the canvas, Monitor and FileBrowser have their own responsive layouts). Body uses `100svh` so the taskbar stays in the always-visible region under the URL bar. The taskbar is horizontally scrollable when narrow, the Start menu spans the screen, and `monitor.html`'s 736px breakpoint collapses its two-column grid into a single scrollable column. xterm.js auto-focus is **suppressed on touch devices** so auto-switching into Terminal doesn't pop up the on-screen keyboard ŌĆö the user taps the terminal to summon it. This has two sources: the desktop's `focus-terminal` `postMessage` (skipped on touch) **and ttyd's own focus when its WebSocket connects** (the one that fires on a fresh load ŌĆö e.g. closing Browser auto-switches to a never-yet-loaded Terminal). The latter is caught by a guard injected into every `/tN/` page via the `sub_filter` in `terminal/install.sh`: on touch, it blurs any focus of the `.xterm-helper-textarea` unless it lands within 700ms of a real touch **inside the terminal page** (a genuine tap whitelists itself in the capture phase; the parent's close-├Ś tap does not). A touch-only **system key bar** (`#sys-keybar`, rendered by `desktop.html` at the top when the keyboard is up ŌĆö the soft keyboard lacks these) gives `esc`/`tab`/`^C`/arrows: each routes `{type:'kbd-key', key}` to the active app (Terminal ŌåÆ PTY bytes via `terminal-kbd.js`; Browser ŌåÆ xpra-patches). The **arrow keys double as a trackpad** ŌĆö a tap sends that arrow once; press-and-**slide** (on any arrow) moves the cursor, distance-proportional (`HSTEP`/`VSTEP` px per step). The slide is **axis-locked** once movement passes 12px. Horizontal slides send **`Ctrl+F`/`Ctrl+B`** (emacs forward/backward-char, `KBD_KEY_BYTES.CtrlF`/`CtrlB` ŌåÆ `\x06`/`\x02`), **not** `ŌåÆ`/`ŌåÉ`: a TUI like Claude Code reuses the *arrows* for menu/history nav on an empty prompt, but Ctrl+F/B move the text cursor and are harmless no-ops when there's nothing to move. Vertical slides send `Ōåæ`/`Ōåō` (no cursor-only equivalent; that axis is history/menu by nature). Touch capture lets the slide continue past the button edge. (This replaced a dedicated `Ōćå` pad key ŌĆö folding it into the arrows saves a slot and reads naturally.) The bar lives at the top on purpose (iOS doesn't report the keyboard height to a nested-iframe input; the strip above the keyboard is unmeasurable), pinned to the visual-viewport bottom via `syncBar`.

### Shared nginx

One nginx site at `/etc/nginx/sites-available/vibetop` (`listen 80 default_server`). The terminal project owns this file. A `map $uri $term_port` directive (generated for 1..50) routes `/tN/` to port `7680+N` via a single regex location block. Sibling projects extend via `include /etc/nginx/snippets/vibetop-extras.d/*.conf`.

### Terminal stack

**Mobile keyboard/dictation** ŌĆö `landing/terminal-kbd.js`, injected into every `/tN/` page via the `sub_filter` `<script src>` (a no-op on non-touch; **desktop keeps native xterm** ŌĆö all keys, tap-to-focus, selection). On touch it lays a **full-height transparent `<textarea>`** over the terminal: tapping it focuses it, so iOS raises the keyboard and **dictation buffers into a real field natively** (like Notes) instead of xterm streaming half-finished revisions to the PTY (the pile-up). Dragging a knob moves that endpoint (clamped so start never crosses end); the drag is **relative** ŌĆö `touchstart` pins the finger's offset from the cell center (`offY`) so it never jumps on grab and is stable when held still. `selStart`/`selEnd` are absolute buffer cells so the handles track scrollback via `positionHandles` on `onScroll` (and skip a handle while it's `_dragging`); a handle scrolled out of view hides. **Re-claim the terminal shape (`terminal-kbd.js`)** ŌĆö terminal N is **one shared `vibetop-session` PTY**, so its `rows├Ścols` belong to whichever device resized last; after you switch active device the TUI inside renders at the *other* device's shape (phone-shaped text on the desktop, desktop-shaped on the phone) ŌĆö and the desktop only auto-re-fits on app *activation*, which misses the case where the phone steals the size while the desktop terminal stays active. A gesture re-sends **this** device's terminal size to the PTY: **double-click** (desktop) / **double-tap** (touch). `claimSize()` sends the resize **straight to ttyd's WebSocket** (`RESIZE_TERMINAL="1"` + `{columns,rows}`, the socket captured by wrapping `window.WebSocket` before ttyd opens it) so the **visible xterm grid never resizes** ŌĆö resizing the grid makes the content visibly jump (the "shake"). But it must **nudge**: ttyd applies each frame via `TIOCSWINSZ`, yet the kernel raises **SIGWINCH only when the size actually changes**, and SIGWINCH is what propagates the size down the chain (`vibetop-session attach`'s SIGWINCH handler ŌåÆ SIGUSR1 ŌåÆ the serve daemon `TIOCSWINSZ`es the shared PTY ŌåÆ SIGWINCHes bash). This client's ttyd PTY is *already* `c├Śr`, so re-sending `c├Śr` is a silent no-op that re-claims nothing ŌĆö so `claimSize` sends a **different size and back**: two real changes ŌåÆ two SIGWINCHes ŌåÆ the shared PTY ends up at this device's shape, with no visible grid resize (the other device then sees mis-shaped output until *it* re-claims ŌĆö symmetric, since the PTY can't be two shapes at once). It nudges the **column, not the row** (`{c-1,r}` then `{c,r}`): the nudge's intermediate size still streams one redraw back to this device, and a row nudge makes a bottom-anchored TUI (a prompt/input box) bounce up a row and back ŌĆö the residual "shake"; a column nudge keeps every row in place (one-column blip for one frame instead). The intermittent shake is then killed outright by a **resize debounce in the `vibetop-session` serve daemon** (`RESIZE_DEBOUNCE` ~35ms): SIGUSR1 arms a deadline instead of resizing inline, and the main loop applies the *latest* size once the burst settles ŌĆö collapsing the nudge's two resizes into a single `TIOCSWINSZ`+SIGWINCH so the shell redraws **once, at the final size** (only **new** sessions get this ŌĆö serve daemons are never restarted). **The shake-fix regression:** an earlier pass sent `c├Śr` *once* over the socket to kill the shake ŌĆö which also killed the re-claim (same-size = no SIGWINCH); the nudge restores both. If the socket wasn't captured it falls back to the old visible `term.resize(c0,r); term.resize(c,r)`. The desktop `dblclickŌåÆclaimSize` handler lives in `terminal-kbd.js`'s non-touch branch (the rest of the file is touch-only), alongside a **Windows Chromium focus fix**: any `term.resize()` ŌĆö the reshape's, the desktop shell's re-fit when the Terminal app is (re)activated/refreshed, or a plain window resize ŌĆö **blurs xterm's hidden input textarea on Windows Chromium and it never refocuses**, leaving the terminal untypable (macOS and touch restore focus themselves). This is what made v1.6.6's reshape break typing the instant you double-clicked to focus the terminal, and a plain refresh leave Windows unable to type, while Mac/iPhone were unaffected. The root fix: re-`term.focus()` right after `term.onResize` (deferred a tick), **guarded by `document.hasFocus()`** so it only refocuses while this page is focused and never steals focus from another app ŌĆö one fix that covers every resize trigger at once. On touch the double-tap is detected by touch **duration** (`<250ms`), **not** `moved` ŌĆö the first tap raises the keyboard and the layout shift while it animates in drifts the finger, which a movement-based check would misread as a scroll and drop the second tap; a brief **"Ōåö resized"** toast confirms it fired, and it blurs the keyboard so reshaping doesn't leave it up.

### Browser stack

One systemd service:
- `vibetop-browser-xpra` ŌĆö xpra `start-desktop :99` with built-in HTML5 client on loopback:14500

xpra handles the virtual X display (Xorg + dummy video driver for RANDR), window management (matchbox in kiosk mode), browser launching (via `browser-loop.sh` wrapper for auto-restart), and the HTML5 client + WebSocket serving. The display dynamically resizes to match the client's browser viewport. Clipboard is handled natively by xpra. xpra is installed from the xpra.org apt repo. `--sharing=yes` lets multiple clients (e.g. desktop + phone) view the same session at once; `XPRA_PING_TIMEOUT=45` (env in the unit) evicts dead clients faster than the 60s default ŌĆö but not lower than 45: phones on power-saving WiFi stall past 20s while alive, and backgrounded Safari tabs stop answering pings, so a 20s timeout evicted live clients.

**Low-bandwidth tuning** (unit encoding flags): targets `quality=80`/`speed=100` but with low floors `min-quality=10`, `min-speed=20`, and `bandwidth-detection=yes` so xpra degrades hard (lower quality, heavier compression) on a constrained link like mobile while a good connection stays sharp. (xpra v6's `--bandwidth-limit` is a fixed bits/sec value, not `auto`; the floors are what let auto-detection actually drop quality.)

nginx proxies `/browser/` to xpra's HTTP/WebSocket port with `sub_filter` patches: CSS pins `#screen` to the viewport via `z-index` (hiding xpra's toolbar/login UI and window-decoration chrome like `.windowhead`/`.window-title` without removing keyboard capture elements like `#pasteboard`), and loads `xpra-patches.js` for mouse offset correction, scroll fix, and **mobile touch handling**. The patches JS file is served from the web root and wrapped in `try/catch` for graceful degradation on xpra updates. A separate regex location caches (`max-age=86400`) and gzips xpra's ~2.1MB HTML5 client assets, which xpra otherwise serves uncompressed and `no-store` ŌĆö the main fix for slow first loads over the tunnel. See `docs/browser.md`.

**Mobile touch in xpra-patches.js** ŌĆö the parent desktop disables iOS pinch-zoom, so all touch gestures are interpreted inside the iframe. A single capture-phase set of `touchstart`/`touchmove`/`touchend` listeners on `window` (xpra binds on `#screen` so we must run first) routes gestures by finger count:
- **1-finger tap (< 10px movement)** ŌåÆ synthetic `mousedown`/`mouseup`/`click` dispatched on the canvas inside `#screen` so xpra forwards a click to the remote.
- **2-finger pinch** ŌåÆ **Safari-style view magnification**: a client-side CSS `transform: translate() scale()` on `#screen` (`VIEWZOOM` state), anchored at the pinch midpoint. The remote layout is **unchanged** ŌĆö it just magnifies the rendered canvas and lets you pan, exactly like Safari zooms a page (an earlier version sent `Ctrl+=`/`Ctrl+-` to Chromium, which reflowed the layout / enlarged fonts ŌĆö not what "zoom" should do). Purely local; the remote never sees it. `getMouse` (patch 1) divides click coords by `VIEWZOOM.z` so taps still land correctly while magnified. Zoom resets to 1├Ś on `orientationchange`. No on-screen zoom buttons ŌĆö pinch out to zoom, pinch back to return to 1├Ś.
- **1-finger drag** ŌåÆ **pan when magnified, scroll at 1├Ś**: zoomed in, the drag pans the CSS-magnified view (clamped to the viewport); at 1├Ś it scrolls the remote page on **both axes** via synthetic `wheel` events on the canvas (vertical `deltaY` + horizontal `deltaX`, accumulated per-axis into `SCROLL_TICK` chunks, default 33, so speed roughly matches the finger).
- **Native keyboard** ŌĆö xpra's drawn `.simple-keyboard` is hidden; instead a real **`<input>` (`#xpra-kbd`) IS the round Ōī© button** on the right edge. This is the crux: **iOS only raises the keyboard when the user taps a real text input directly** ŌĆö programmatically `focus()`-ing a separate hidden input (from a `<div>` button, or from a canvas tap) does *not* work. So the input itself is the tap target (transparent text/caret, `font-size:16px` to avoid focus-zoom; the Ōī©/Ō£Ģ glyph is a `pointer-events:none` sibling overlay). Tapping it focuses it ŌåÆ keyboard; a second tap (when already focused) blurs to dismiss (`pointerdown` + `activeElement` check). The window-capture touch layer (patch 4) **ignores taps on `#vkb-toggle`** (`onChip` guard) so the input's native focus isn't prevented. Typed characters are forwarded to the remote as synthetic key events (xpra reads `event.code`/`key`/`keyCode`; its handlers don't check `isTrusted`), built by **diffing the input's value** on each `input` event so backspace and autocorrect replacements work; Enter/Tab/empty-backspace go via `keydown`.
- **Paste on non-Mac** ŌĆö xpra uses Meta as its clipboard modifier on macOS but Control elsewhere, via the browser `paste` event, which is unreliable on Windows (Cmd+V works, Ctrl+V often doesn't). On non-Mac the patch intercepts `Ctrl+V` (capture phase, before xpra's document handler), reads the local clipboard, and types it into the remote via the same synthetic-key path. The working Mac Cmd+V path is untouched.
- **Auto-reconnect (never show the disconnect screen)** ŌĆö xpra's HTML5 client only auto-reconnects on *abnormal* WS close codes (`1006/1008/1010/1014/1015`); a **clean close (1000)** ŌĆö iOS suspending a backgrounded tab, or the server's `XPRA_PING_TIMEOUT` eviction ŌĆö does **not** reconnect, so xpra's "connection lost" page sticks. The patch listens for xpra's own `connection-lost`/`connection-established` document events: on lost it arms a 2.5s timer to `location.reload()` `/browser/` (reconnects fresh; the remote session/windows are untouched), cancelled if `connection-established` fires first (xpra's own reconnect won). Reload is deferred while the tab is hidden (retried on `visibilitychange`) and floored to once per 8s via `sessionStorage` so a genuinely-down server can't reload-loop. This is the Browser analogue of the ttyd "Press ŌÅÄ to Reconnect" guard.
- **Re-claim the display size (`xpra-patches.js` patch 10)** ŌĆö the Browser is a **single shared** xpra `start-desktop :99` display, so its resolution belongs to whichever client connected/resized last: when the phone connects it advertises its small viewport (`_screen_resized()` ŌåÆ `configure_display {"desktop-size":[ŌĆ”]}`) and the display shrinks for **every** client, incl. the desktop. xpra's `_screen_resized()` re-sends only when **this** client's `container` actually changed, so the desktop (whose window didn't change) can't grow it back on its own ŌĆö same "same-size = no-op" wall as the Terminal re-claim. Fix (the Browser analogue of the Terminal's double-click/tap "claim the shape"): a **desktop double-click** *or* **mobile double-tap** busts the guard (`client.desktop_width = -1`) and calls `client._screen_resized()`, re-sending this client's real container size so the server RANDR-resizes the display to match. It re-claims **only when the display size differs from our viewport** ŌĆö **two-directional** (the desktop grows it back after a phone shrank it; the phone shrinks it after the desktop grew it ŌĆö a "smaller than me" test is a no-op on the phone, whose display is *bigger*), measured from the largest `id_to_window`'s `w`/`h` ŌĆö so an ordinary double-click/tap doesn't spam RANDR resizes, and it **never `preventDefault`s** so word-select still reaches the remote. Desktop is detected from `pointerdown` timing (capture phase, independent of xpra/jQuery); the touch double-tap is detected in the patch-4 touch layer's `touchend` tap branch (two no-movement taps within 400ms/28px, additive to the taps already sent). Manual + symmetric like the Terminal ŌĆö the other device sees this one's size until *it* re-claims (a shared display can only be one size).

### Update

A self-update app at `/update.html` (Start menu ŌåÆ **System** section) that pulls the latest commit from GitHub and redeploys, driven by the manager. `GET /api/update` shows the installed commit/date/subject **plus a `history` array** ŌĆö the **real per-host self-update log** for THIS deployment (not the git changelog): events `deployed` (a baseline seeded with the current commit on the manager's first start Ōēł deploy time), `updated` (with `from`/`to` and the list of commits actually pulled), and `failed`. Newest-first, persisted at `~APP_USER/.local/share/vibetop-update-history.json` (capped at 200). The app renders it as an **Update history** list with a **Clear** button ŌåÆ `POST /api/update/history/clear` (writes `[]`). `POST /api/update`:
1. `git fetch` then fast-forward to `origin/main` in `REPO_DIR` (the manager's own checkout, `<repo>/terminal/terminal-manager.py` ŌåÆ `<repo>`). Git runs as `APP_USER` via **`sudo -u APP_USER -H`** (not `su -`, which would print the host MOTD banner into the output) ŌĆö the repo owner, who also holds the GitHub SSH key (root would trip git's "dubious ownership" guard). Non-interactive SSH must work (passphraseless key / agent). **Dirty-tree handling:** a plain `git pull --ff-only` aborts if the working tree is dirty ŌĆö which happens when a host was deployed by **rsync** (`deploy.sh --remote` / manual rsync copies files in *without* committing). The dirtiness check is `git status --porcelain --untracked-files=no` ŌĆö **untracked files are ignored**: they never block a fast-forward, and counting them made the next check (`git diff --quiet origin/main`) always read as "local edits" whenever the host was *behind* (the tracked tree always differs from upstream then), so a host with any untracked cruft could never self-update. For genuine **tracked** dirtiness the manager checks: if the tree already **matches `origin/main`** (`git diff --quiet origin/main` ŌĆö the rsync case, changes are redundant), it `git reset --hard origin/main`; if the changes are **genuine local edits not upstream**, it bails ŌĆö returning `blocked:"dirty"` plus the file list so `update.html` shows them with a **"Discard local changes & update"** button. That button re-POSTs `{force:true}`, which `git stash push --include-untracked` (recoverable via `git stash list`/`pop` on the host ŌĆö *not* a destructive `reset --hard`) and then fast-forwards. Upshot: prefer the in-app Update (or `git pull` on the host) over rsync for sharing committed work; rsync-deployed hosts self-heal on the next Update, and stray edits can be cleared self-service from the app.
2. If the pull brought changes, redeploy **only what changed** (by `git diff --name-only`): `landing/` ŌåÆ `landing/install.sh` (as `APP_USER`); `browser/` ŌåÆ `browser/install.sh`; `terminal/` ŌåÆ `terminal/install.sh` (regenerates the nginx site); `office/` ŌåÆ `office/install.sh` with `INSTALL_CONTAINER=0` (just re-renders the `/onlyoffice/` nginx snippet ŌĆö the live OnlyOffice container is **left running**; container arg/image changes need a full deploy, same as systemd-unit changes below); `files/` **or `landing/filebrowser-patches.js`** ŌåÆ `files/install.sh` (re-renders the `/files/` snippet ŌĆö needed because that patch JS lives under `landing/` but its `?v=` cache-buster is computed by `files/install.sh`, so a patch-only change must refresh the snippet or the browser keeps the stale cached JS). The browser/terminal/office/files scripts run **as root** (the manager is root) with `APP_USER=ŌĆ” INSTALL_DEPS=0 INSTALL_SYSTEMD=0` so they only redeploy files + reload nginx.
3. If a manager module changed ŌĆö **any `.py` directly under `terminal/`** (`terminal-manager.py` or a sibling like `system_status.py`; `tests/` and the path-independent `vibetop-session` are excluded) ŌĆö the manager restarts **out-of-band** via `systemd-run --on-active=3 systemctl restart vibetop-manager` (a transient timer, so the restart survives the manager's own death ŌĆö a child in the manager's cgroup would be killed mid-restart). The response is sent first; `update.html` re-polls the version after the blip.

The whole log (each step's stdout/stderr + ok/fail) is returned and shown in the app. **After a successful update that changed something, the app reloads the whole desktop** (the top window, after a short countdown ŌĆö waiting for the API to come back if the manager restarted) so the new cached shell + service worker take effect; nothing reloads if already up to date. **Bootstrap:** the *first* deploy of this feature needs a manual `sudo systemctl restart vibetop-manager` (the running manager predates the `/api/update` route); after that it self-updates.

### Claude plan-usage strip (`claude-usage/`, opt-in)

A thin full-width strip at the **top** of the desktop showing the **real** Claude Max-plan usage ŌĆö `session (5h) NN% ┬Ę resets Xh ┬Ę week (7d) NN%`, color-graded (green <70 / amber <90 / red). **There is no API to query plan usage**; the numbers exist only as `anthropic-ratelimit-unified-*` **response headers** on live API calls (`ŌĆ”-5h-utilization`/`-5h-reset`/`-7d-utilization`/`-representative-claim`). ### Services dashboard (`landing/index.html`, `terminal/service_discovery.py`)

The **Services** app (served at `/landing.html`) ŌĆö an auto-discovering list of the host's network services. `service_discovery.py` is a stdlib sibling module (like `system_status.py`/`claude_stats.py`) the manager imports; `GET /api/services/discover` calls `service_discovery.discover()`, memoized ~5s. It runs `ss -H -tlnp`, and for each listener bound to a non-loopback address that survives the port/proc denylists, classifies it via `/proc/<pid>/cmdline`:
- **Filtering** ŌĆö loopback binds (`127.*`/`::1`) are dropped (excludes all of vibetop's own internal ports), as are infra ports (`_SKIP_PORTS`: ssh/smtp/dns/rpc/mail/db + vibetop's own `:80`) and backend daemons (`_SKIP_PROCS`). Because a daemon run under a generic interpreter reports the *interpreter* as its `ss` proc name (`python3 ŌĆ”/vsmagent`), `_effective_proc` pulls the real script/program token from the cmdline for both filtering and naming.
- **Naming** ŌĆö `classify()`: a `_RECOGNIZERS` regex table (Open WebUI, Ollama, ThinLinc `tlwebaccess`/`tlwebadm`, Jupyter, Streamlit, Grafana, code-server, Plex, Home Assistant, ŌĆ”, incl. which are HTTPS) ŌåÆ a `_PORT_HINTS` fallback (e.g. 8501ŌåÆStreamlit, 11434ŌåÆOllama) ŌåÆ `<proc> :<port>`.
- **Output** ŌĆö `{lan_ip, services:[{name, desc, port, url, proc, health}]}`; `url` is `scheme://<lan_ip>:<port>/` (lan_ip = the default-route source addr), `health` is `"up"` (it's listening ŌåÆ green dot).

The page (`index.html`) fetches `/api/services/discover` directly (LAN) or takes the parent's `postMessage` relay (tunnel), renders a **3-per-row** card grid (single column Ōēż640px), and on a card tap **opens the service in a normal new browser tab** (`<a target="_blank">`) ŌĆö the right target for a service on your own LAN (full browser, no xpra latency). A secondary **Ō¦ē Browser** button opens it in the **embedded Browser** instead (`open-in-browser` `postMessage`; Chromium is on the host, so it reaches the LAN service even through the tunnel where the LAN address isn't directly reachable ŌĆö falls back to a direct `POST /api/browser/open` when standalone). (An earlier version made the card body itself open the embedded Browser; that was demoted after the embedded hand-off proved flaky ŌĆö see the `RestrictNamespaces` entry in `docs/design-decisions.md`.) Pure helpers (`parse_ss`/`classify`/`_effective_proc`) are unit-tested in `tests/test_service_discovery.py`.

### URL forwarding (terminal/files ŌåÆ browser)

Clicking a URL in a terminal (Cmd+click / Ctrl+click) or using the "Open in Browser" action in Files opens it in the embedded Chromium. Implementation:
- nginx `sub_filter` injects a `window.open` override into terminal pages
- The override intercepts xterm.js's link handler (which calls `window.open()` then sets `.location.href`) and returns a proxy object
- The proxy sends the URL to `POST /api/browser/open`, which runs `chromium <url>` on the xpra display
- The manager passes `DBUS_SESSION_BUS_ADDRESS` and `--user-data-dir` matching the xpra profile so the URL opens in the correct Chromium instance
- A `postMessage` to the parent desktop auto-switches to the Browser tab
- Files opened from FileBrowser use the `/fileview/` nginx location (alias to `~`) to serve raw files to Chromium

### Tunnel

`cloudflared` maintains an outbound connection to Cloudflare. All traffic for `service.example.com` routes to `localhost:80`. Cloudflare Access handles auth (email PIN + Google). Config lives at `/etc/cloudflared/config.yml` (rendered from `tunnel/config.yml.template`).

## Install script conventions

All `install.sh` scripts share the same patterns:
- Idempotent and re-runnable. `--dry-run` (or `-n`) previews without acting.
- Env vars override defaults (e.g. `MAX_INSTANCES=50`, `XPRA_PORT=14500`). See the header comment in each script for the full list.
- Systemd unit files under `*/systemd/` are templates with `@PLACEHOLDER@` tokens (e.g. `@APP_USER@`, `@DISPLAY_NUM@`). install.sh renders them via `sed` and writes to `/etc/systemd/system/`.
- nginx configs under `*/nginx/` follow the same pattern.

## Versioning & commits

- The root **`VERSION`** file (e.g. `1.5.6`) is a hand-maintained release number ŌĆö it is **not** read by any build/deploy script (the `VERSION` strings in `browser/install.sh`/`files/install.sh` are the OS `VERSION_CODENAME` and `FB_VERSION`, unrelated). Bump it when cutting a user-visible release and commit with a `vX.Y.Z: <summary>` subject (see `git log -- VERSION`).
- **Shell/static-page changes** that affect the cached PWA shell must bump the service-worker cache version: `VERSION` in `landing/sw.js`. The convention is a `(sw vNN->vNN)` suffix on the commit subject (e.g. `ŌĆ” (sw v47->v48)`). Sub-resource JS injected via `sub_filter` (`xpra-patches.js`, `filebrowser-patches.js`, `terminal-kbd.js`) is content-hash cache-busted automatically ŌĆö never bump those by hand.
- Per-sub-project `CLAUDE.md` files (`terminal/`, `browser/`, `landing/`) are thin pointers back to this root file and `docs/` ŌĆö the canonical architecture lives here; keep edits here, not duplicated there.
- **Release checklist (order matters):** 1) bump `VERSION` **and** `landing/sw.js`'s `VERSION` (if the cached shell changed), 2) **commit + push**, 3) **then deploy** ŌĆö via the in-app **Update** app (it `git pull`s origin/main, redeploys only what changed, and bumps the *deployed* `~/vibetop-www/sw.js`) or `./deploy.sh`. **A git push alone does nothing to a running host** (no webhook/CI auto-deploy), and the SSE auto-refresh (`/api/events`) only pushes a `reload` to open clients when the **deployed** `sw.js` VERSION changes ŌĆö so **deploying *before* the sw bump is committed** (or never deploying after a bump) leaves the deployed copy stale and no client auto-refreshes. If you ever bump-then-deploy out of order, just redeploy landing (`./landing/install.sh`) to push the new version out.

## Gotchas

- Snap chromium can't use `--user-data-dir` outside its confinement ŌĆö the xpra profile lives at `~/snap/chromium/common/xpra-profile`. The `/api/browser/open` handler must pass this profile and `DBUS_SESSION_BUS_ADDRESS=unix:path=/run/user/<uid>/bus` when running `chromium` via `su`, otherwise the URL silently goes nowhere ("Opening in existing browser session" fails without D-Bus, and without `--user-data-dir` it targets the wrong Chromium instance)
- **`vibetop-manager.service` must NOT set `RestrictNamespaces`** ŌĆö snap Chromium/Firefox launch via `snap-confine`, which creates a mount namespace; that directive blocks it for the service and all its children, so `/api/browser/open` (terminal/Files/Services link-open) and the X11 Launcher's snap apps **silently fail to start** (`Popen` still returns "ok"). Symptom: the Browser app switches into view but the page never loads. It's dormant until a manager **restart** reloads the unit (hence "worked until it was restarted"). Kept off on purpose alongside `NoNewPrivileges`/`ProtectHome`/`PrivateTmp`. Full write-up in `docs/design-decisions.md`
- **Snap chromium needs lingering enabled for `APP_USER`** (`browser/install.sh` runs `loginctl enable-linger` in its systemd step). The xpra service runs as `APP_USER` with no login session; snap launches chromium inside a transient user scope (`snap.chromium.chromium-<uuid>.scope`), which requires the user's systemd instance + `$XDG_RUNTIME_DIR` (`/run/user/<uid>`). Without lingering, systemd-logind tears down `/run/user/<uid>` once the deploying login session ends, so every chromium launch fails ŌĆö the journal shows `is not a snap cgroup for tag snap.chromium.chromium` (snapd) and `Failed to create secure directory (/run/user/<uid>/pulse)` (xpra) on a 2s loop while `browser-loop.sh` crash-loops, and the Browser app stays **blank**. It "recovers" the instant any login session for that user appears (e.g. an SSH login recreates `/run/user/<uid>`), which masks the bug. Fix on an affected host: `sudo loginctl enable-linger <user>` then `sudo systemctl restart vibetop-browser-xpra`
- xpra is installed from xpra.org's apt repo (GPG key at `/usr/share/keyrings/xpra.asc`, source at `/etc/apt/sources.list.d/xpra.sources`) ŌĆö Ubuntu's packaged v3.1.5 is too old (no HTML5 client)
- The browser-loop.sh wrapper (at `/usr/local/lib/vibetop-browser/browser-loop.sh`) auto-restarts chromium on crash ŌĆö xpra's `--start-child` only runs the command once
- Desktop page at `~/vibetop-www/index.html` is `landing/desktop.html`; old landing page preserved at `~/vibetop-www/landing.html`
- `www-data` gets traversal on `/home/myuser` via ACL (`setfacl`), preserving the home dir's 0750 mode
- xterm.js scrollback requires `sub_filter` injection in nginx ŌĆö ttyd 1.7.4's runtime option setter doesn't resize the buffer. The same filter injects other xterm options into ttyd's config literal: `sub_filter 'fontSize:13,' 'fontSize:13,scrollback:$SCROLLBACK,cursorStyle:"bar",cursorInactiveStyle:"bar",'` ŌĆö so the cursor is a **bar** (vertical line), not the default block. **Both** keys are needed: `cursorStyle` is the focused cursor, but this xterm build defaults `cursorInactiveStyle` to `"outline"` (a hollow box) whenever the terminal loses focus ŌĆö which, nested inside the desktop iframes, is most of the time; setting it to `"bar"` too keeps the line in both states (the "cursor still a box" symptom = only `cursorStyle` was set). Adding an xterm option = extend that one replacement (no ttyd restart; `/tN/` is `no-store` so a reload picks it up)
- Clipboard in terminals uses a DOM-based copy technique injected via nginx `sub_filter`: on **Ctrl+C/Cmd+C with a selection** the selected text is written into xterm.js's helper textarea, selected via `ta.select()`, then copied via `document.execCommand('copy')` (and the key does NOT send SIGINT). This is the only approach that works inside nested iframes ŌĆö the async Clipboard API (`navigator.clipboard.writeText`) and `clipboardData.setData()` in copy event handlers are both silently blocked by browsers in iframe contexts. Copy is **explicit** (this keystroke or native right-click) ŌĆö there is **no copy-on-select**: auto-copying on every selection silently clobbered the clipboard whenever you highlighted text just to read it. **Two** copy-on-select paths had to be killed: our own `onSelectionChange` hook (removed), **and ttyd's built-in one** ŌĆö ttyd registers its own `onSelectionChange` that does `document.execCommand("copy")` then `overlayAddon.showOverlay("Ō£é",200)` (a scissors overlay), with no flag to disable it. It's neutralized by **gating `document.execCommand`**: an override injected at the top of the sub_filter **throws on a `"copy"` command unless `window.__allowCopy` is set**, so ttyd's handler throws inside its own `try/catch` and bails **before both the copy and the Ō£é** ŌĆö while our explicit copies (Cmd/Ctrl+C's `copySelection`, the touch Copy button's `execCopy`) set `window.__allowCopy=true` around their `execCommand` and go through. Native right-click Copy is unaffected (it fires a `copy` *event*, not `execCommand`). (Matches the touch flow, which copies via an explicit long-press ŌåÆ Copy button.) **Paste:** Mac Cmd+V pastes natively. On non-Mac, the `customKeyEventHandler` **returns false for Ctrl+V** so xterm never emits a literal `^V` ŌĆö which a TUI like Claude Code reads as a paste-image keystroke ("no image in clipboard"). The actual paste then rides xterm's **own `paste` event** ŌĆö the exact path right-click-paste uses ŌĆö so it works in both secure and insecure (HTTP-LAN) origins with no Clipboard API. (An earlier attempt called `navigator.clipboard.readText()`, but that object is `undefined` on an HTTP origin, so the handler fell through to `return true` and `^V` was sent.) The `/tN/` page is served `Cache-Control: no-store` so a stale browser cache can't silently keep old sub_filter behaviour after an update
- All desktop iframes use `allow="clipboard-read; clipboard-write"` ŌĆö without this, clipboard paste into the xpra browser iframe fails
- Terminal iframes in `terminals.html` also need `allow="clipboard-read; clipboard-write"` for clipboard to work through the double-nested iframe chain (desktop ŌåÆ terminals ŌåÆ /tN/)
- `vibetop-session` attach client clears `ICRNL` from terminal input flags so `\r` (Enter) passes through to TUI apps like Claude Code without being converted to `\n`. Does NOT use full `tty.setraw()` to preserve output post-processing
- xpra CSS uses `z-index` overlay (not `display:none`) to hide the toolbar ŌĆö hiding body children with `display:none` breaks keyboard input because xpra's `#pasteboard` textarea (used for keyboard capture) must remain in the DOM
- **xpra "new computer can't type in the Browser (mouse works)"** ŌĆö the HTML5 client sends its keymap on connect as `["keymap-changed", {keymap}, false]`; that trailing `false` is the server's `force` flag. xpra's long-lived `start-desktop` session keeps the keyboard config from the *first* client that ever connected, and since every HTML5 client hashes to the same keymap, a later client (different computer/session) matches and is **skipped** (`keyboard mapping already configured (skipped)` in the xpra log) ŌĆö its keys translate against the stale config and are dropped, so the mouse works but typing does nothing. Fix is **patch 7** in `xpra-patches.js`: wrap `XpraClient.prototype.send` and flip the `keymap-changed` packet's force flag to `true`, so the server re-applies the keymap for whichever client is connecting.
- **Cache-busters are content-hash-derived ŌĆö never bump a `?vN` by hand.** The three nginx-`sub_filter`-injected scripts use a `?v=@PATCH_VER@` placeholder in their config templates; the installer computes `md5sum | cut -c1-10` of the JS and seds it in: `xpra-patches.js` (`browser/install.sh`ŌåÆ`browser/nginx/browser.conf`), `filebrowser-patches.js` (`files/install.sh`ŌåÆ`files/nginx/filebrowser.conf` ├Ś2), `terminal-kbd.js` (`terminal/install.sh`, interpolated into the site config as `${KBD_VER}`). Editing any of these files changes its hash ŌåÆ the `?v=` changes ŌåÆ nginx + the service worker fetch the new copy, and `nginx_write` sees the changed config and reloads. This makes "forgot to bump the version ŌåÆ shipped stale JS" structurally impossible. The shell pages (`desktop.html` etc.) are still covered by `sw.js` `VERSION` (bump that for shell/static-page edits).
- **Service worker only caches known shell pages.** `sw.js` caches a *navigation* only when its pathname is in `SHELL_PAGES` (derived from `PRECACHE`: `/`, `landing.html`, `notes.html`, `monitor.html`, `upload.html`); any other HTML (`office-editor.html`, `update.html`, `loggedout.html`, ŌĆ”) is **network-only**, so it can never be served stale after a deploy that didn't bump `VERSION`. Sub-resources (JS/CSS/icons) are cache-first/SWR keyed by full URL incl. `?v=` ŌĆö so the content-hash cache-busters above invalidate them correctly.
- **ttyd waits for the session socket before serving** (`vibetop-ttyd@.service` `ExecStartPre`). `After=`/`Requires=` only order process *start*, not socket readiness; without the wait, on a cold start (first terminal after a reboot/reset) ttyd could accept a WebSocket and spawn `vibetop-session attach` before the daemon bound `/tmp/vibetop-session-N.sock`, failing the first connection. Because ttyd serves HTTP only after `ExecStartPre`, `terminals.html`'s `waitReady()` (polls `/tN/` for HTTP 200) now also implies the backend is ready.
- **Manager state files are written atomically** (`_atomic_write`: temp in the same dir + `os.replace`) for `desktop-state.json`, the update-history, and `desktop-notes.md`. A plain `open('w')`+write can be observed mid-write by a concurrent reader (the registry GET/POST run on multiple `ThreadingHTTPServer` threads) ŌĆö for the desktop registry that would reset `reset_epoch` and drop every instance's state. Update-history read-modify-write is additionally serialized by `_update_lock`; office save-back uses a per-thread temp name so concurrent OnlyOffice callbacks can't corrupt the file.
- **Desktop iframe activation messaging is deferred to the frame's `load`** (`loadIfNeeded`ŌåÆ`notifyActiveFrame` in `desktop.html`). On first open the frame's `src` was just set, so `focus()`/`postMessage`/`resize`/`focus-terminal` fired synchronously would land in `about:blank` and be lost ŌĆö the "first open doesn't focus / misses its `vibetop:active` broadcast" race. The deferred notify is guarded so a frame that finishes loading after the user switched away doesn't steal focus.
- xpra patches (`browser/xpra-patches.js`) are served as a standalone JS file from the web root, wrapped in `try/catch` for graceful degradation if xpra updates change the API
- System status API auto-detects the discrete GPU by picking the card with the most VRAM. CPU temp from `k10temp` (hwmon), GPU temp from `amdgpu` edge sensor (hwmon). During heavy GPU compute, the driver locks sysfs files (`EBUSY`); util/temp/power then fall back to parsing `/sys/kernel/debug/dri/N/amdgpu_pm_info` ("GPU Load", "GPU Temperature", "W (average SoC|GPU)") so the numbers stay populated. The manager runs as root, which is required for debugfs (0700)
- **The status collection itself lives in `terminal/system_status.py`**, a sibling module the manager imports (`system_status.get_system_status(running_terminals, cached)`), not in `terminal-manager.py`. The CPU/RAPL/disk/process snapshot globals moved with it; the manager injects its running-terminal list and the shared `_cached` memoizer (which stays in the main module because terminal start/stop invalidates its `running_terminals` entry). The manager runs **in-place from the git checkout** (`ExecStart=ŌĆ”/terminal-manager.py`), so the sibling import needs no install change ŌĆö but the self-update restart trigger in `_handle_update` treats **any `.py` directly under `terminal/`** (not just `terminal-manager.py`) as a manager module, so a pulled change to `system_status.py` still restarts the API
- Terminal manager API validates URLs for `/api/browser/open` by rejecting shell metacharacters to prevent command injection
- Terminal instances are dynamic (on-demand via `/api/terminals/`). Systemd template units are not pre-enabled; the manager API starts/stops them. Only `vibetop-manager.service` is enabled at boot
- **Moving/renaming the repo directory requires re-rendering the 3 systemd units' `@APP_DIR@` AND restarting `vibetop-ttyd@N`.** `ttyd-run.sh` bakes the absolute `vibetop-session` path into ttyd's per-connection spawn command (`${SCRIPT_DIR}/vibetop-session attach N`) at ttyd start time, so a running ttyd keeps spawning the *old* path on every fresh WebSocket connect ŌĆö which fails instantly and shows as a terminal that flashes "connected" in a reconnect loop (existing connections survive because their attach client was already spawned). The `vibetop-session serve` daemons are path-independent at runtime ŌĆö they `execv('/bin/bash')` for shells and serve `/tmp/vibetop-session-N.sock` ŌĆö so they keep working and must NOT be restarted (that would kill the live shells). So after a move: fix the units, `systemctl restart vibetop-ttyd@N` (sessions preserved), and restart `vibetop-manager` (its `REPO_DIR` is computed from its own script path)
- The `window.open` override in terminal pages intercepts xterm.js link clicks by returning a proxy object with a setter on `.location.href` ŌĆö this matches xterm.js's pattern of `window.open()` then `obj.location.href = url`
- nginx uses a `map $uri $term_port` directive (generated for 1..50) and one regex location block instead of per-instance location blocks
- FileBrowser config is in `~/.config/filebrowser/filebrowser.db` (set via `filebrowser config set`), not CLI flags
- FileBrowser patches (`landing/filebrowser-patches.js`) are loaded via nginx `sub_filter` injection, not bundled with FileBrowser. The patches hide Vue's conditional action buttons and replace them with permanent always-visible buttons ŌĆö CSS for these must be scoped to `.fb-permanent` to avoid overriding the `display:none` hiding of Vue's buttons. On desktop widths (>736px), `#dropdown` content renders inline in the header (not as a popup), so dropdown buttons also need label styling
- `landing/install.sh` must run without `sudo` ŌĆö with `sudo`, `$HOME` resolves to `/root/` and files deploy to the wrong directory
- tunnel's `install.sh` only installs the `cloudflared` binary ŌĆö tunnel creation and Access setup are interactive (see `tunnel/README.md`)
- xpra is single-client by default ŌĆö without `--sharing=yes`, opening the Browser tab on a second device evicts the first, and a desktop+phone pair kick each other in a loop so the phone "never loads." Cloudflare-tunnel clients arrive from `127.0.0.1`, so in xpra logs a remote tunnel client appears as a loopback address (not the real client IP)
- xpra's HTML5 client assets are served `no-store` and uncompressed; the `/browser/` nginx snippet adds a regex asset location that caches + gzips them. nginx won't gzip proxied responses without `gzip_proxied any`, and xpra's JS Content-Type is `text/javascript` (not `application/javascript`) ŌĆö both must be set or compression silently no-ops
- Desktop iframes get their `src` set on first activation (`loadIfNeeded`), not at iframe-creation time. ttyd inside an iframe whose ancestor is `display:none` initialises xterm.js with a 0├Ś0 viewport, leading to truncated history that only a refresh fixes; deferring the `src` lets the inner content measure the real viewport on the first paint
- **Clickable chrome next to an app iframe ŌĆö two recurring traps** (these are runtime/rendering bugs that static review misses; verify by actually clicking in a browser): (1) **paint order** ŌĆö an app iframe is a *positioned* element (`position:absolute` in a `position:relative` container), so it paints **above** a static-positioned tab/taskbar and steals clicks where they overlap. Give the bar `position:relative` + a higher `z-index` (terminal `.tabs` z-index:2 over `.frames`; the desktop `.taskbar` is already z-index:100). Symptom: "only the upper part of the tab/button responds." (2) **focused-iframe swallows the first click** ŌĆö when an app iframe has focus (you've been typing), the browser spends the first click on the parent moving focus back to the window, so `click`-based activation needs two clicks. Bind activation on **`pointerdown`** for mouse instead (keep `click` for close buttons and for touch, where pointerdown would hijack a scroll-drag). Both fixes are in the terminal tab bar (`terminals.html`) and the desktop taskbar (`desktop.html`).
- `/api/upload` uses a hand-rolled streaming multipart parser (`_BoundaryReader` + `_iter_multipart_files`). Reads from `self.rfile` are capped by `_LimitedReader(rfile, Content-Length)` because once the body is exhausted, a bare `rfile.read(n)` blocks forever on a keep-alive socket waiting for n more bytes
- iOS pinch-zoom is page-wide and irreversibly sticky once activated, so the desktop shell's viewport meta uses `user-scalable=no,maximum-scale=1`; per-app zooming (xpra canvas, FileBrowser UI) is the inner app's responsibility. Each app's iframe handles its own scroll on `100svh` body so the taskbar stays in the always-visible viewport region
- xpra's HTML5 client renders its own window chrome (`.windowhead`/`.window-title`/`.title-bar`) where the WM name ("matchbox") would otherwise be visible ŌĆö these selectors are added to the CSS hide list in the `/browser/` nginx sub_filter
- **Dual-homed deploy host (two NICs on the SAME LAN subnet) breaks terminal/Browser WebSockets** ŌĆö if the host has both ethernet and WiFi (or any two NICs) up on the *same* subnet, terminals connect then drop on a fixed **~10s cycle** ("can't type"; ttyd logs `WS closed from 127.0.0.1` every ~10s, reconnect-loop) for **some** LAN clients but not others ŌĆö while a **loopback** WebSocket and **another single-NIC host running the same build** are fine. This is **not a vibetop bug** (reverting versions changes nothing ŌĆö it reproduced identically on an older release) and **not the client** (the same laptop works against a single-NIC host); it's host networking. The host answers ARP for *both* IPs on *both* NICs (**ARP flux**) and routes *all* replies out the lowest-metric interface, so a connection that **arrives** on the other NIC gets **asymmetric** replies that die at the TCP/WS keepalive timeout. It's **path-dependent**, so it hits some devices and not others depending on which IP/MAC each resolved (e.g. Apple mDNS tends to land on the primary/ethernet IP ŌåÆ works; a client pinned to the secondary IP ŌåÆ flaps) ŌĆö which masks it as a browser/OS/app problem and sends you chasing the wrong layer. **Diagnose:** `ip -brief addr` + `ip route` showing two IPs / two default routes on one subnet; a loopback WS survives but a LAN-client WS flaps. **Fix:** `deploy.sh` **auto-handles this** ŌĆö it detects 2+ NICs on one subnet and runs `tools/setup-samesubnet-routing.sh` (no-op on single-homed hosts), so a dual-homed host works with the normal deploy and no manual host networking. The script stops the ARP flux (`arp_ignore=1`/`arp_announce=2`) and installs per-interface source routing (one routing table per NIC, each defaulting out its own interface, so replies leave on the **incoming NIC**) via a NetworkManager dispatcher that re-applies on every up/dhcp event and **auto-detects IP/subnet/gateway** ŌĆö no hardcoded addresses, pure `iproute2` (no iptables). Idempotent + network-agnostic; needs NetworkManager for boot persistence. Alternatively single-home the host (`nmcli radio wifi off`) ŌĆö and the truly tweak-free options (different subnets, failover-only) are in `docs/dual-homed-network.md`. Full cross-platform write-up ŌĆö incl. how **Windows** behaves (it mostly avoids this by default via the strong host model) and how to do dual networking with **no** OS tweaks ŌĆö is in `docs/dual-homed-network.md`.
